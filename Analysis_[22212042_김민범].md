# 1. Introduction

### 1.1 문서 개요 (Executive Summary)
본 Analysis 문서는 대한민국 1인 가구 비중이 36.1%에 육박하는 인구 구조 변화 속에서, 대학생 및 자취생의 식비 부담을 체계적으로 관리하기 위한 **H.O.M.E(Hidden Opportunity Meal Economics)** 시스템의 상세 분석을 목적으로 한다. 기존의 식비 관리 방식이 단순 지출액 기록에만 머물렀다면, 본 시스템은 **기회비용(Opportunity Cost)**과 **가사 노동 비용(Labor Cost)**을 통합한 총비용 모델을 제시한다. 이를 통해 사용자는 '배달, 밀키트, 직접 조리'라는 세 가지 선택지 사이에서 자신의 시간 가치를 반영한 최적의 경제적 의사결정을 내릴 수 있다.

### 1.2 시스템의 유용성 (Usefulness)
* **실질 비용 왜곡 해소:** 프랜차이즈의 매장가 대비 높은 배달 전용 가격과 숨겨진 배달팁 구조를 가시화하여, 소비자가 표시 가격만으로 의사결정하는 비용 착시 현상을 제거한다.
* **개인 맞춤형 노동 비용 반영:** 사용자의 실제 가사 노동 시간과 설거지 도구 사용량에 따른 비용을 정량화하여, '직접 해 먹으면 무조건 싸다'는 고정관념에서 벗어난 정밀한 비교를 가능하게 한다.
* **지속 가능한 절약 동기 부여:** 'Hidden Opportunity Report' 기능을 통해 월간 누적 절약액을 시각화하고, 단순 수치가 아닌 실질적인 경제적 가치로 환산하여 장기적인 합리적 소비 습관 형성을 지원한다.

### 1.3 시스템의 중요성 및 차별성 (Significance)
* **기회비용 계수(0.2) 도입:** 조리 중 TV 시청·음악 감상 등 멀티태스킹이 가능하다는 점을 반영하여, 순수 조리 시간의 20%만 기회비용으로 산정하는 독창적인 비용 모델을 적용한다. 이는 직접 조리의 시간 비용을 과대 산정하지 않으면서도 체계적으로 반영하는 균형 잡힌 접근이다.
* **2026년 최저임금 기준 적용:** 시급 **10,320원**을 기준으로 시간 가치를 산정하여 분석의 현시성과 정확도를 확보하며, 사용자가 직접 수정할 수 있도록 설계하여 개인 상황에 따른 유연한 적용이 가능하다.
* **실시간 표준 가격 연산 알고리즘:** 마트의 묶음 판매 단위와 메뉴별 필요량 매핑의 복잡성을 해결하기 위해, 시스템은 **네이버 쇼핑 API를 통해 조회된 재료별 표준 최저가 시세를 실시간으로 매핑하여 합산하는 알고리즘**을 적용한다. 이를 통해 복잡한 매몰 비용(Sunk Cost) 오류를 방지하고 사용자가 실제로 체감하는 식재료비 총합을 주도적으로 산출한다.

### 1.4 발전 및 확장성 (Expandability)
* **식재료 폐기 비용(Disposal Cost) 추적:** 유통기한 관리 기능을 추가하여 사후 발생하는 폐기 비용을 Net Saving에서 차감함으로써, 월간 절약액 산출의 정확도를 고도화할 수 있다.
* **잔여 재료 기반 메뉴 추천:** 단위가격 환산 이후 남은 재료 데이터를 누적하여, 다음 끼니 선택 시 냉장고 내 잔여 재료를 우선 활용하는 메뉴를 제안하는 방향으로 확장 가능하다. 이는 폐기 비용 절감과도 직접 연결되어 시스템의 핵심 목표를 강화한다.
* **메뉴 데이터베이스 확장:** 초기 버전에서는 카테고리별 대표 메뉴 3~5개 지원에서 시작하여, 사용자 요청 및 데이터 누적을 바탕으로 지원 메뉴를 점진적으로 확대할 수 있다. 이를 통해 더 다양한 식사 상황에서 비용 비교 분석이 가능해진다.

# 3. Domain Analysis

아래의 그림은 H.O.M.E 시스템의 Domain Analysis에서 도출된 핵심 클래스들의 관계를 나타낸 것이다.

<img src="./images/09_domain_diagram.png" width="300">

H.O.M.E 시스템은 크게 **사용자 및 메뉴 관리**, **외부 데이터 수집**, **비용 계산**, **결과 기록 및 리포트** 네 가지 역할 그룹으로 클래스를 분류할 수 있다. 각 클래스의 역할과 관계는 다음과 같다.

## 3.1 클래스 설명

### 1) User
사용자의 계정 정보와 로그인 상태, 월간 절약 로그를 관리하는 클래스이다. 사용자는 `Menu` 클래스를 통해 분석할 메뉴를 선택하고, `VirtualMartDB`를 통해 배달 가격 정보를 조회하는 흐름의 시작점이 된다.

**주요 속성 및 메서드:**
* 속성 `userId`: 사용자 고유 식별자
* 속성 `loginStatus`: 로그인 상태 (true / false)
* 속성 `savingLog`: 누적 절약 로그 목록 (`AnalysisLog` 참조)
* 메서드 `login()`: 계정 인증 및 세션 시작
* 메서드 `logout()`: 세션 종료
* 메서드 `viewReport()`: 월간 Hidden Opportunity Report 조회 요청

### 2) Menu
분석 대상 메뉴의 이름, 카테고리, 조리 가능 여부를 관리하는 클래스이다. `menuType` 속성을 통해 배달·밀키트·직접 조리 세 옵션 중 어떤 옵션을 사용자에게 제공할지 결정한다.

**주요 속성 및 메서드:**
* 속성 `menuName`: 메뉴 이름 (예: 제육볶음)
* 속성 `category`: 음식 카테고리 (한식 / 일식 / 중식 / 양식 / 프랜차이즈)
* 속성 `menuType`: 조리 가능 여부 (`ALL` / `NO_COOKING` / `DELIVERY_ONLY`)
* 메서드 `getAvailableOptions()`: menuType에 따라 활성화할 옵션 목록 반환

### 3) VirtualMartDB
배달의민족 기준으로 수동 수집한 배달 가격 데이터베이스 클래스이다. 메뉴명으로 조회하면 배달 음식 가격과 최소주문금액을 반환한다. 배달팁은 사용자가 UC-03에서 직접 입력하므로 본 클래스에는 포함되지 않는다.

**주요 속성 및 메서드:**
* 속성 `menuName`: 메뉴 이름
* 속성 `foodPrice`: 배달 음식 가격 (원)
* 속성 `minOrder`: 최소주문금액 (원)
* 메서드 `lookup(menuName)`: 메뉴명으로 배달 가격 정보 조회

**DB 예시:**
| menu_name | food_price | min_order |
|:---:|:---:|:---:|
| 제육볶음 | 9,000원 | 12,000원 |
| 된장찌개 | 8,000원 | 10,000원 |

### 4) ExternalApiService
농촌진흥청 조리식품 레시피 DB API 및 네이버 쇼핑 오픈 API 연동을 총괄 정의하며, 선택된 메뉴의 재료 목록, 재료별 사용량(g/ml), 조리 시간(min)을 수집 및 정형화하여 반환하는 클래스이다. API 미등록 메뉴 발생 시 폴백 전략(사용자 입력 → 카테고리 평균 상수)을 적용한다.

**주요 속성 및 메서드:**
* 속성 `recipeApiKey`: 농촌진흥청 API 인증키
* 속성 `shoppingApiKey`: 네이버 쇼핑 API 인증키
* 속성 `fallbackTimeMap`: 카테고리별 평균 조리 시간 상수 테이블
* 메서드 `fetchRecipe(menuName)`: 메뉴명으로 레시피 데이터 조회 및 반환
* 메서드 `applyFallback(category)`: API 미등록 시 카테고리 평균 상수 반환

**fallbackTimeMap 기본 조리 시간 상수:**
| 카테고리 | 직접 조리 | 밀키트 |
|:---:|:---:|:---:|
| 한식 | 35분 | 20분 |
| 일식 | 25분 | 15분 |
| 중식 | 30분 | 15분 |
| 양식 | 30분 | 20분 |
| 프랜차이즈 | — | 10분 |

### 5) Ingredient (DTO)
`ExternalApiService`와 `ShoppingService` 사이를 연결하는 **DTO(Data Transfer Object)** 역할의 클래스이다. 레시피 API에서 반환된 재료 목록과 네이버 쇼핑 API에서 조회한 가격 데이터를 하나의 객체로 통합하여 `CostCalculator`에 전달한다.

> **아키텍처 최적화 반영:** 시스템 가용성 및 런타임 안정성을 극대화하기 위해, 본 클래스는 순수 데이터 보관 및 전달 역할(POJO DTO)에 집중하도록 설계하였다. 이에 따라 복잡한 연산 메서드 및 유도 속성은 데이터베이스와 메모리 낭비를 방지하고자 클래스 내부에서 영속화하지 않고, 핵심 비즈니스 엔진인 **`CostCalculator` 레이어의 로직 스트림 내부에서 실시간 인라인 연산**되도록 구조를 최적화하였다.

**주요 속성 명세:**
* 속성 `ingredientName`: 재료명 (레시피 API 반환 및 정규화 테이블 필터링)
* 속성 `salePrice`: 네이버 쇼핑 오픈 API 기반 실시간 판매 최저가
* 속성 `totalVolume`: 해당 식재료 상품의 마트 표준 규격 총용량 (g 또는 ml)

**실시간 수식 연산 및 데이터 흐름 예시 (CostCalculator 엔진 내 가동):**
* `단위가격(unitPrice) = salePrice ÷ totalVolume`
* `실사용 비용(actualCost) = 단위가격 × 레시피 표준 사용량(recipeUsage)`

| ingredientName | salePrice | totalVolume | recipeUsage | unitPrice | actualCost |
|:---:|:---:|:---:|:---:|:---:|:---:|
| 돼지고기 앞다리 | 8,000원 | 500g | 200g | 16원/g | 3,200원 |
| 진간장 | 3,000원 | 500ml | 15ml | 6원/ml | 90원 |
| 참기름 | 6,000원 | 160ml | 5ml | 37.5원/ml | 187원 |

### 6) ShoppingService
네이버 쇼핑 API를 통해 식재료의 실시간 최저가와 판매 단위(g/ml), 밀키트 판매 가격을 수집하는 클래스이다. 식품 카테고리 필터를 적용하여 비식품 상품(씨앗, 화분 등)이 최저가로 잘못 매핑되는 오류를 방지한다.

**주요 속성 및 메서드:**
* 속성 `apiKey`: 네이버 쇼핑 API 인증키
* 속성 `categoryFilter`: 식품 카테고리 코드 (비식품 매핑 차단)
* 메서드 `fetchPrice(ingredientName)`: 재료명으로 최저가 및 판매 단위 조회
* 메서드 `fetchMealKitPrice(menuName)`: 메뉴명으로 밀키트 판매 가격 조회

### 7) IngredientMapper
`ExternalApiService` 내부에서 데이터 가공을 수행하는 유틸 성격의 정규화 매핑 테이블 클래스이다. 초기 버전에서는 30~50개 항목을 수동으로 구축하며, 매핑 실패 시 유사어 검색 후 수동 입력 폴백을 적용한다.

**주요 속성 및 메서드:**
* 속성 `mappingTable`: 레시피 재료명 → 쇼핑 검색어 매핑 딕셔너리
* 메서드 `normalize(rawName)`: 원본 재료명을 쇼핑 검색어로 변환
* 메서드 `addMapping(rawName, searchTerm)`: 새 매핑 항목 추가

**매핑 테이블 예시:**
| 레시피 재료명 (rawName) | 쇼핑 검색어 (searchTerm) |
|:---|:---|
| 대파 | 대파 1단 |
| 돼지앞다리살 | 돼지고기 앞다리 |
| 참기름 | 참기름 식용 |
| 간장 | 진간장 |

### 8) CostCalculator
H.O.M.E 시스템의 핵심 계산 클래스이다. `ExternalApiService`, `ShoppingService`, `VirtualMartDB`로부터 수집한 데이터를 바탕으로 배달·밀키트·직접 조리 세 옵션의 총비용을 산출하고, 최적 선택과 Net Saving을 반환한다.

**주요 속성 및 메서드:**
* 속성 `minWage`: 최저임금 시급 (기본값: 10,320원, 사용자 수정 가능)
* 메서드 `calcDelivery(foodPrice, deliveryFee, minOrder)`: 배달 총비용 산출
* 메서드 `calcMealKit(kitPrice, cookMin)`: 밀키트 총비용 산출
* 메서드 `calcCooking(ingredientCost, cookMin, laborMin, toolCost)`: 직접 조리 총비용 산출
* 메서드 `getBestOption()`: 최솟값 옵션 및 Net Saving 반환
* 메서드 `calcUnitPrice(salePrice, totalVolume, usage)`: 재료 단위가격 환산

**핵심 비용 공식:**
* 배달: `foodPrice + deliveryFee + max(0, minOrder − foodPrice)`
* 밀키트: `kitPrice + (cookMin ÷ 60) × 0.2 × minWage`
* 직접 조리: `Σ(salePrice_i ÷ totalVol_i × usage_i) + (cookMin ÷ 60) × 0.2 × minWage + (laborMin ÷ 60) × minWage + toolCost`
* 단위가격: `salePrice ÷ totalVolume`
* Net Saving: `배달 총비용 − min(배달, 밀키트, 직접 조리)`

### 9) AnalysisLog
매 분석 완료 시 생성되는 절약 로그 클래스이다. 분석 날짜, 메뉴명, 사용자 선택 옵션, 배달 기준가 대비 절약액을 저장하며, 월간 리포트 집계의 원천 데이터로 활용된다.

**주요 속성 및 메서드:**
* 속성 `date`: 분석 날짜 (YYYY-MM-DD)
* 속성 `menuName`: 분석 메뉴명
* 속성 `chosenOption`: 사용자가 선택한 옵션 (배달 / 밀키트 / 직접 조리)
* 속성 `deliveryBaseCost`: 배달 기준 총비용
* 속성 `actualCost`: 실제 선택 옵션의 총비용
* 속성 `saving`: 절약액 (`deliveryBaseCost − actualCost`)
* 메서드 `save()`: 서버 User DB에 로그 저장

### 10) ReportGenerator
`AnalysisLog` 데이터를 집계하여 월간 Hidden Opportunity Report를 생성하는 클래스이다. 누적 절약액을 시각화하고, 절약액의 실질적 활용 방안을 제안하는 동기부여 메시지를 함께 생성한다.

**주요 속성 및 메서드:**
* 속성 `targetMonth`: 리포트 대상 월
* 메서드 `generateMonthlyReport(userId, month)`: 월간 절약 리포트 생성
* 메서드 `calcMonthlySaving(logs)`: 월간 누적 절약액 집계
* 메서드 `getSuggestionMessage(totalSaving)`: 절약액 기반 활용 제안 메시지 생성

## 4.5 비용 정보 입력 화면
<img src="./images/05_input.png" width="380">

선택한 메뉴(예: 김치찌개·한식)에 대해 **세 가지 방식의 비용 파라미터**를 수집하는 화면. 단순 가격 비교를 넘어 **기회비용(Opportunity Cost)** 까지 산출하기 위한 핵심 단계다.

| 섹션 | 주요 입력 항목 | 비고 |
|------|--------------|------|
| 🔴 배달 정보 | 배달 음식 가격, 배달팁, 최소주문금액 | 배달팁은 구독 서비스(배민클럽 등) 고려해 가변 인자로 처리 |
| 🟡 밀키트 정보 | 밀키트 가격, 조리 시간 | 가격은 API로 실시간 최저가 자동 로드 |
| 🟢 직접 조리 정보 | 조리 시간, 설거지 시간, 사용 도구 선택 | 조리 시간은 API 자동 입력 |

**도구 선택 및 단가:**
* 프라이팬 200원 / 냄비 200원 / 수저 100원 / 가위 100원 / 집게 100원
* 하단 **도구 비용 합계** 실시간 계산 표시 (예: 300원)

> 설거지 시간과 도구 단가는 가사 노동력과 세제·수도세를 수치화한 핵심 변수.

## 4.7 세부 비용 내역 화면 (Result Breakdown Specification)
사용자가 산출된 결과의 정밀한 근거를 신뢰하고 체감할 수 있도록, 백엔드 엔진(`CostCalculator`)의 연산 결과 도출 스냅샷을 항목별로 투명하게 공개하는 인터페이스 명세이다. 프론트엔드의 `ResultScreen` 하단 컴포넌트와 백엔드 DTO 객체가 일대일 거울 구조로 매핑된다.

### 1) 화면 인터페이스 레이아웃 및 항목 명세
화면 하단에 배치되는 세부 명세는 옵션별 활성화 상태에 따라 동적으로 바인딩되며, 2026년 최저임금 기준 시급 10,320원을 고정 기준으로 명시하여 데이터의 현시성을 보장한다.

* **🛵 배달(Delivery) 세부 명세 항목**
  * **음식 가격 (`foodPrice`):** 선택한 배달 메뉴의 기본 단가 매핑
  * **배달팁 (`deliveryFee`):** 사용자가 설정한 목적지 기준 배달 수수료
  * **최소주문금액 보정 (`Math.max(0, minOrder - foodPrice)`):** 최소주문 보증을 채우기 위해 추가 지출해야 하는 경제적 비용

* **📦 밀키트(Mealkit) 세부 명세 항목**
  * **밀키트 가격 (`kitPrice`):** 시중 밀키트 조리 상품의 표준 온라인 소비자가
  * **기회비용 (`Math.round((kitMin / 60) * 0.2 * MIN_WAGE)`):** 분 단위 밀키트 조리 시간에 대한 가치 손실분 

* **🍳 직접 조리(Cooking) 세부 명세 항목**
  * **식재료비 (`ingredientCost`):** 오픈 API에서 실시간 쿼리로 조회된 원재료 상품 표준 판매가의 합산 주입값
  * **기회비용 (`Math.round((cookMin / 60) * 0.2 * MIN_WAGE)`):** 순수 요리 진행 시간에 따른 기회비용 
  * **가사노동비 (`Math.round((laborMin / 60) * MIN_WAGE)`):** 조리 후 설거지 및 가사 노동 정리에 투입된 온전한 시간 가치
  * **도구 비용 (`toolCost`):** 선택된 프라이팬, 냄비 등 소모 비용 합계

## 4.8 Hidden Opportunity Report (절약 리포트)
<img src="./images/08_report.png" width="250">

사용자가 시스템을 지속적으로 사용하도록 유도하는 **Gamification 요소이자 누적 성과 리포트**이다.

**설계 의도:** 배달 대비 절약 금액을 전광판 형태로 시각화하여 사용자가 '돈을 벌고 있다'는 긍정적인 경험을 제공한다. 절약 금액에 따른 동기부여 메시지와 시각적 아이콘(💪, 💡)을 통해 합리적인 식습관 형성을 유도한다.

* 이번 달 **총 절약액 5,868원** 전광판 형태로 강조 표시
* 동기부여 메시지: *"💡 조금씩 쌓이고 있어요! 계속해봐요 💪"*
* **분석 내역 History 로그** — 날짜별 소비 패턴 회고 가능
  * 예: 김치찌개 / 직접 조리 선택 → **+5,868원 절약** / 4,132원 지출
* **새 분석 시작하기 →** 버튼으로 재분석 진행

# 5. Glossary

| 용어 | 정의 |
|------|------|
| 가사 노동 비용 | 설거지 및 조리 후 정리 등 식사 준비 과정에서 발생하는 노동 시간의 경제적 가치 |
| 기회비용 | 조리 및 식사 준비 시간 동안 다른 생산적인 활동을 하지 못함에 따라 발생하는 잠재적 비용 |
| 기회비용 계수 | 조리 중 멀티태스킹(음악 감상 등) 가능성을 고려하여 실제 시간에 적용하는 가중치 (0.2) |
| 단위가격 | 판매 단위와 레시피 사용량 간의 차이를 해결하기 위해 판매가 ÷ 총용량으로 환산한 1단위당 가격 |
| 데이터베이스 | 사용자 계정, 배달 메뉴 가격 정보 및 누적 절약 로그가 저장되는 가상의 저장소 |
| 메뉴타입 | 조리 가능 여부에 따라 메뉴를 분류하는 속성 (`ALL`, `NO_COOKING`, `DELIVERY_ONLY`) |
| 밀키트 | 손질된 식재료와 양념이 세트로 구성되어 간단한 조리 과정만 거치면 되는 식사 옵션 |
| 배달팁 | 배달 서비스 이용 시 음식 가격 외에 추가로 지불하는 배송 비용 |
| 사용자 (자취생) | H.O.M.E 시스템을 통해 식비 비용을 분석하고 합리적인 소비 결정을 내리는 주체 |
| 절약액 (Net Saving) | 배달 총비용 대비 밀키트나 직접 조리를 선택했을 때 실질적으로 아낀 금액 |
| 카테고리 | 한식, 일식 등 음식의 특성에 따라 메뉴를 구분하는 분류 체계 |
| 팝업창 | 예외 상황 알림이나 데이터 입력 확인 등을 위해 화면 위에 일시적으로 나타나는 UI 요소 |
| DTO | Data Transfer Object의 약자. 시스템 계층 간에 데이터 영속성 없이 네트워크 수송만을 전담하는 POJO 객체 |
| Entity | 데이터베이스 테이블과 직접 지도로 매핑되어 비즈니스 정체성과 생명주기를 영속화하는 자바 도메인 클래스 |
| Use Case | 시스템과 외부 액터 간의 상호작용 흐름을 명세하여 시스템이 제공하는 독립적 기능 단위를 명세화한 기획 도구 |
| Sunk Cost | 이미 지출되어 현재나 미래의 의사결정 과정에서 회수 및 고려 대상에서 완전히 제외해야 하는 매몰 비용 |
